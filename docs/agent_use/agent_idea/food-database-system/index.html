使用情境
三種模式，覆蓋所有進廚房的理由
節源、規劃、探索——各有不同的輸入與輸出邏輯
1
節源模式 — 冰箱剩什麼，就做什麼
冰箱打開，不知道要吃什麼，就讓系統告訴你。
使用者操作
勾選冰箱食材：紅蘿蔔、高麗菜、蛋
勾選：我想看能做什麼
系統回應
紅蘿蔔炒蛋 ✓ 完全符合
菇菇炒高麗菜 ✓ 完全符合
2
買買買模式 — 這週想吃什麼，一次規劃
先選這週想吃什麼，系統計算所有需要的食材，一次購物清單。
使用者操作
勾選這週想吃的：滷肉飯、咖哩飯、煎餃
我有這些庫存：紅蘿蔔、高麗菜
系統回應
購物清單： 里肌肉、滷蛋、滷豆腐 海帶、咖哩塊、馬鈴薯 蔥、蒜、煎餃皮 （已扣除庫存品項）
3
AI對話模式 — 不知道吃什麼，讓 AI 幫你想
不知道吃什麼，AI 根據你的冰箱庫存推薦，並引導你探索新選擇。
我不知道要吃什麼
那你冰箱有什麼？
高麗菜、紅蘿蔔、蛋、火鍋料
不如今天吃火鍋搭配蛋花湯怎麼樣？
但我想多買一點
好的！來挑挑有什麼喜歡吃的吧（系統根據冰箱已有食材，展開相關推薦）
勾選：麻辣鍋、炸蛋、滷味拼盤
採買建議： 鴨血、豆腐、麻辣湯底 炸蛋食材、滷味配料 （火鍋料家裡已有）
資料庫架構
8 個子資料庫，撐起整個系統
每個資料庫解決一個問題，分工明確、可以各自擴充
食材庫
每種食材的名稱、營養素、GI值、發炎指標、保存方式、保存期限
核心顆粒度
食材狀態庫
使用者冰箱的目前庫存：數量、購買日期、過期預估、存放位置（冷藏/冷凍）
冰箱掃描
食譜庫
每道菜的完整配方：所需食材列表、步驟、整備時間、烹飪時間、困難度
推薦主體
Expand 關聯庫
每道菜的「相關菜」清單：使用相同核心食材的其他料理、用戶歷史做過的延伸
Expand 引擎
食材價格庫
每種食材的市場零售價格（每日更新）、批发價、特價週期、替代食材建議
成本控制
餐點成本計算
每道菜的食材成本加總 + 人力水電估算 -> 顯示真實成本與自己煮省了多少
省錢感知
餐廳外食庫
各餐廳的菜單、價格、食材組成、地點、等候時間（懶得做時的外食替代）
外食替代
使用者偏好學習
使用者喜歡/不喜歡的食材、做過的菜、歷史評分、近期購買記錄（Local 儲存）
個人化排序
Expand 推薦引擎
核心邏輯：食材 Expandability
不是「你有什麼食材能做什麼菜」，而是「你選了這道菜，系統幫你展開相關的所有可能」

Expand 運作範例

用戶勾選
紅蘿蔔、高麗菜
展開 →
相關食譜：
紅蘿蔔炒蛋、滷肉飯
菇菇炒高麗菜、炸豬排
✓ = 庫存已有
用戶選了
滷肉飯

相關推薦：紅燒肉、咖哩飯
（用相同蛋白質来源）
→ 再 Expand

炸豬排

麵包粉 ✗
✗ = 需購買

推薦維度
三個維度同時評估，決定推薦排序
系統同時計算這三個分數，加權後輸出最終排序
📦
庫存覆蓋度
這道菜有多少%食材家裡已經有？100% = 完全符合，不需要買
🔗
Expandability
選這道之後，相關延伸料理有多少%家裡也有？越高越容易持續擴展
⏰
時間窗口
整備時間 + 烹飪時間，符合使用者現在的時間嗎？

維度加權（系統預設，可由使用者調整）
