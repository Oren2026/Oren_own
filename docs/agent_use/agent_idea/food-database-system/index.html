資料庫架構
8 個子資料庫，撐起整個系統
每個資料庫解決一個問題，分工明確、可以各自擴充
食材庫
每種食材的名稱、營養素、GI值、發炎指標、保存方式、保存期限
核心顆粒度
食材狀態庫
使用者冰箱裡每項食材的目前庫存：數量、購買日期、過期預估、存放位置
冰箱掃描
食譜庫
每道菜的完整配方：所需食材列表、步驟、整備時間、烹飪時間、困難度
推薦主體
時機標籤庫
每道菜的時機標籤：減脂/增肌/宵夜/省錢/抗發炎/助眠/高蛋白，每個標籤對應評分權重
推薦維度
食材價格庫
每種食材的市場零售價格（每日更新）、批发價、特價週期、替代食材建議
成本控制
餐點成本計算
每道菜的食材成本加總 + 人力水電估算 -> 顯示真實成本與自己煮省了多少
省錢感知
餐廳外食庫
各餐廳的菜單、價格、食材組成、地點、等候時間
外食替代
使用者偏好學習
使用者喜歡/不喜歡的食材、做過的菜、歷史評分、近期購買記錄
個人化排序
推薦引擎
match 分數背後的邏輯
不是單純的「有/沒有」，而是多維度的符合度評估

評估維度（同時計算）

食材符合度
這道菜需要的食材，我冰箱裡有多少%符合？完全符合=100分
時間符合度
這道菜的整備時間，符合使用者現在的時間窗口嗎？
成本符合度
食材成本 + 額外購買成本，符合今天預算嗎？
時機標籤符合
這道菜的標籤與使用者當下需求的符合程度
歷史偏好
這道菜的歷史評分，影響排序位置

輸出結果

完全符合（綠色）
家裡有的食材，100% 覆蓋這道菜的需求。可以立刻做，不需要額外採買。
部分符合（黃色）
缺 1-2 樣食材。系統同時列出：需要買什麼、哪家有特價、預估多花多少錢。
外食替代（紫色）
如果懶得做或來不及，系統推薦附近可外送的餐廳。
