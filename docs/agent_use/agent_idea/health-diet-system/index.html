運動 × 飲食
雙資料庫整合系統
打造一個能根據個人身體狀態、健身進度、
情緒與目標，動態推薦訓練與餐點的個人化引擎
健身資料庫 食物資料庫 AI 推薦引擎 精誠資料庫整合
系統願景
解決什麼問題？
現有健康 APP 的最大弱點：給你一個制式計劃，然後你自己看著辦。
📋
現有方案的問題
計畫制式化，不考慮疲勞、睡眠、情緒。
食物推薦與訓練完全脫節。
執行意願低，因為吃不到喜歡的。
💡
我們的切入點
把「健身」和「飲食」當成同一個系統的兩個變數。
動態調整：狀態差就減量，狀態好就加碼。
食物永遠是快樂的來源，不是罪惡感。
核心邏輯
客戶狀態
→
姿勢圖譜 + 訓練模型
→
食譜 + 食材 + 餐廳
→
Top3 推薦
輸入：「最近睡眠不好，但這週要備戰 TDK」→ AI 回覆：「禮拜一輕量訓練＋助眠食材，週末衝刺」
健身資料庫
7 個子資料庫，構成身體模型
每筆資料都服務「什麼狀態，該做什麼訓練」的推薦決策
個人檔案
體重、體脂、肌肉量、健身目標、飲食限制、過敏原、復原能力
基本屬性
姿勢圖譜
動作名稱、視覺示範影片、適用肌群、設備需求、動作危險性評估
動作庫
肌群地圖
每個肌群包含哪些動作、負責什麼功能、與其他肌群的連動關係
肌肉力學
姿勢變化庫
同一動作的不同角度、握法、器械差異、替換動作建議
動作變體
訓練效果模型
組數 × 次數 × 重量 → 訓練量（Volume）、疲勞度指數、恢復所需時間
推薦核心
恢復追蹤
每次訓練後的疲勞感、睡眠品質、肌肉痠痛程度、心率變異（HRV）
動態調整
偏好學習
哪些動作喜歡/討厭、進步最快的動作、穩定進步的動作模式
個人化
食物資料庫
8 個子資料庫，構成飲食模型
讓「吃什麼」不只是熱量計算，而是身體狀態的響應
食譜庫
配方、做法、整備時間、營養成分（熱量/蛋白質/碳水/脂肪）、標籤（增肌/減脂/助眠/抗發炎）
內容主體
食材庫
每種食材的營養素、GI 值、發炎指標、對血糖的影響、適合什麼時機吃
顆粒度
食材價格趨勢
每日/每週市場價格、替代食材建議、CP 值最高的時機
成本控制
餐點成本計算
食材成本 + 人力 + 店家利潤 = 真實報價基礎
報價用
餐廳菜單庫
各店家的餐點名稱、價格、食材組成（用於評估是否符合需求）
外部串接
餐廳評價庫
口味分數、環境分數、服務分數、使用者回饋標籤（方便/貴/配料多）
信任分數
地點服務
餐廳位置座標、外送覆蓋範圍、等候時間、預約需求
即時性
時機推薦庫
什麼時間/狀態吃什麼（運動前/後、睡前、壓力大、生理期、康復期）
推薦核心

輸出：兩種回答模式

🏋️
訓練建議
「今天適合練什麼、幾組、幾次」
包含動作順序、替代方案、預期疲勞度
🍽️
餐點推薦
「今天運動完適合吃什麼」
包含自煮選項、外送選項、附近餐廳推薦
實作方向
