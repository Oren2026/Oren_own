# Deadlock（死結）

## 定義

死結（Deadlock）是多執行緒環境中的一種「僵局」。當兩個或多個執行緒都在**等待對方釋放資源**，導致所有參與者都無法繼續執行時，就發生了死結。

想像一個情境：哲學家就餐問題。
執行緒 A 拿著左手的筷子，等待右手的筷子；執行緒 B 拿著右手的筷子，等待左手的筷子。兩個人都不肯放手，結果兩人都餓死。

---

## 1. 死結發生的四個必要條件

根據 Coffman 準則，死結發生**必須同時滿足**以下四個條件。只要打破其中一個，就能避免死結：

| 條件 | 說明 |
|------|------|
| **互斥 (Mutual Exclusion)** | 資源一次只能被一個執行緒使用（如 Mutex 本身） |
| **持有並等待 (Hold and Wait)** | 執行緒已經持有至少一個資源，但又在申請其他被佔用的資源 |
| **不可搶奪 (No Preemption)** | 資源不能被強制剝奪，只能由持有者主動釋放 |
| **循環等待 (Circular Wait)** | 存在一個執行緒循環鏈，每個執行緒都在等待下一個執行緒所持有的資源 |

---

## 2. 如何避免死結？

### 固定加鎖順序（最常用）

強制規定所有執行緒必須**按照相同的順序獲取鎖**。例如，不論是執行緒 1 還是 2，都必須「先鎖 A、再鎖 B」。這樣執行緒 2 就不會在持有 B 的情況下去等 A，從而打破了「循環等待」。

### 使用 trylock

如果拿不到鎖，就立即返回錯誤而不是死等。這時執行緒可以先釋放掉自己手上已有的鎖，過一陣子再重試。

### 逾時機制

設定等待上限，超時就放棄並報錯。

---

## 3. 如何偵測死結？

如果你的程式突然「停住」且 CPU 使用率極低，通常就是死結。在 Linux 上，你可以使用 gdb 附加到進程，查看各個執行緒的堆疊追蹤（Backtrace），觀察它們是否都卡在 `pthread_mutex_lock`。

---

## 總結

死結就像是程式碼裡的「交通堵塞」，通常源於過於複雜的資源依賴。設計時遵守「一致的加鎖順序」是最簡單有效的防範原則。
