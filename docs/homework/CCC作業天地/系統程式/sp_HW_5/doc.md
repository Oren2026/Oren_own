# 實作說明文件

## 銀行存提款模擬（bank_simulation）

### 概念
本程式用於演示 Race Condition（競態條件）對共享資料造成的影響，並展示 Mutex（互斥鎖）如何解決此問題。

### 實作方式
- 使用 JavaScript Async/Await 模擬執行緒切換
- 臨界區分為三步：Read（讀取）→ Modify（修改）→ Write（回寫）
- `useMutex = false` 時，兩執行緒可同時進入臨界區，導致讀取到相同的余額值，最後一回寫覆蓋前一結果
- `useMutex = true` 時，透過上鎖確保同一時間只有一個執行緒在操作余額

### 關鍵程式碼邏輯
```
READ:  balance = 1000   (Thread A)
READ:  balance = 1000   (Thread B)  ← 同時讀到相同值
MOD:   new = 999         (Thread A)
WRITE: balance = 999    (Thread A)
MOD:   new = 999         (Thread B)
WRITE: balance = 999    (Thread B)  ← 覆蓋了 Thread A 的結果！
```

### 執行方式
- 打開 `bank_simulation/index.html`
- 選擇「不使用 Mutex」，點「開始模擬」
- 觀察：提款 50000 次 + 存款 50000 次，最終余額不等於初始值 $1,000

## 生產者消費者問題（producer_consumer）

### 概念
經典的同步問題。共享緩衝區（大小為 5），生產者放入資料，消費者取出資料。需要同時處理：
- **互斥（Mutual Exclusion）**：同時只有一個執行緒能操作緩衝區索引
- **同步（Synchronization）**：生產者不能放進滿的緩衝區，消費者不能取出空的緩衝區

### 實作方式
- **empty Semaphore**：空位計數，初始值 = 5
- **full Semaphore**：資料計數，初始值 = 0
- **mutex**：保護緩衝區索引（in / out）
- 操作順序：先 `sem_wait(&empty)` 再 `pthread_mutex_lock(&mutex)`（順序錯誤會死結！）

### 為何順序重要？
如果先鎖 mutex 再等 sem_wait：當緩衝區滿時，生產者拿著 mutex 等空位，但消費者因為拿不到 mutex 無法取出資料，永遠不會有空位 → 死結。

## 哲學家用餐問題（dining_philosophers）

### 概念
五位哲學家坐在圓桌，五支筷子放在相鄰哲學家之間。每位哲學家需要拿到左右兩支筷子才能吃飯。

### 死結的形成
當所有哲學家**同時**執行以下動作：
1. 拿起左邊的筷子
2. 嘗試拿右邊的筷子

此時：每支筷子都被鄰居拿著，沒有人能拿到第二支筷子，行程式碼層面的「交通堵塞」——**死結（Deadlock）**。

### 四個 Coffman 條件（死結必要條件）
| 條件 | 滿足原因 |
|------|---------|
| 互斥 | 筷子一次只能被一個哲學家持有 |
| 持有並等待 | 拿了左邊筷子後，等右邊 |
| 不可搶奪 | 別人不能搶走已拿到的筷子 |
| 循環等待 | P0等P1的筷子，P1等P2的...P4等P0 |

### 解決方案：固定加鎖順序
規定：所有哲學家**按照相同的順序拿筷子**（先拿編號小的）。
- P0：先拿筷子0
- P1：先拿筷子0（已被P0拿走，等！）
- ...

如此形成的是**鎖鏈（Chain）**而非**循環（Circular）**，打破 Coffman 第四條件。

### 無順序模式（會死結）
在「無順序」模式下，點擊「自動執行」數次後，系統會檢測到所有哲學家都在等待，宣布死結。

## 使用的技術

| 項目 | 技術 |
|------|------|
| 前端頁面 | HTML5 + CSS3 + Vanilla JS |
