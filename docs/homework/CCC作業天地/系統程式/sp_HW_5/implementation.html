銀行存提款模擬

問題情境

兩個執行緒同時操作同一個銀行帳戶：執行緒 A 存款 $2、執行緒 B 存款 $1，各執行 100,000 次。

Race Condition 原理

// 無 Mutex：兩執行緒同時讀到同余額，後寫入覆蓋先寫入 // 執行緒 A // 執行緒 B READ(balance) // $0 READ($0)（同時讀到！） MODIFY($0+$2=$2) MODIFY($0+$1=$1) WRITE($2) // 余額=$2 WRITE($1)（覆蓋！）

每筆操作損失 $1（A 的 $2 被 B 的 $1 覆蓋），100,000 筆後累積誤差約 $100,000。

實作架構

模式邏輯
無 MutexA 和 B 各自讀到 balance_before，然後各自寫入，後寫入覆蓋先寫入
有 MutexA 完整執行完後才釋放鎖，B 才能讀到 A 寫入後的值

快速執行模式

單步執行適合教學觀察，但 100,000 次太慢。快速執行使用 for 迴圈直接跑完所有操作，最後一次性更新 UI 並顯示統計結果。

哲學家用餐問題

問題情境

5 位哲學家坐在圓桌周圍，每人之間有一根筷子。哲學家需要同時拿到左右兩根筷子才能開始吃，吃完後放下兩根筷子，鄰居才能拿。

死結形成

哲學家0：拿到 筷子1，等待 筷子0 哲學家1：拿到 筷子2，等待 筷子1 → 形成循環 哲學家2：拿到 筷子3，等待 筷子2 哲學家3：拿到 筷子4，等待 筷子3 哲學家4：拿到 筷子0，等待 筷子4 ← 循環封閉 所有人都拿了一根，都等待另一根，永遠吃不到。

實作架構

狀態程式邏輯
飢餓（Hungry）嘗試取得左右筷子
吃飯中（Eating）同時拿到兩根筷子
思考中（Thinking）已放下筷子

筷子邏輯

// 第 i 位哲學家的左右筷子 left(i) = chopsticks[(i - 1 + 5) % 5] // 左邊筷子（i-1） right(i) = chopsticks[i] // 右邊筷子（i）

解決死結

讓至少一位哲學家改變拿筷子的順序（先拿左再拿右），打破循環等待。

視覺呈現

// CSS transform 極座標定位 phil-slot: rotate(θ) translateY(-160px) rotate(-θ) chop-slot: rotate(θ) translateY(-135px) rotate(-θ)

θ 為該位置的極座標角度（每格 72°），translateY 距離決定半徑，內層恢復角度確保元素不旋轉。

生產者消費者問題

問題情境

生產者把物品放進緩衝區（上限 N 個），消費者從緩衝區取出物品。緩衝區為空時消費者必須等待；緩衝區滿時生產者必須等待。

同步條件

條件保護什麼
互斥（Mutual Exclusion）緩衝區同一時間只有一個執行緒能操作
同步（Synchronization）空/滿的狀態必須正確傳遞

實作架構

使用 setTimeout 模擬非同步的放入/取出操作。緩衝區內容用顏色方塊表示，狀態面板顯示：

  • 緩衝區已使用格數 / 總格數
  • 各執行緒目前狀態（運行中 / 等待中）
  • 已生產總量 / 已消費總量

技術棧

所有模擬均為純 HTML + CSS + JavaScript，單一檔案，無需任何框架或依賴：

優點 ✅

直接在瀏覽器開啟即可運行

適合教學展示

GitHub Pages 直接托管

無需 npm install / build

限制 ❌

JavaScript 是單執行緒

無法真正模擬多核心平行

但足以演示邏輯問題：

Race Condition 的核心是「時序不確定」，即使單核 CPU 的排程切換也會造成同樣問題
