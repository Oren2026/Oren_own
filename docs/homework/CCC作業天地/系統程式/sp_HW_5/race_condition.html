⚠️ Race Condition 競態條件

當多個執行緒同時存取並試圖修改共享資料，而最終結果取決於操作執行的先後順序時，就會發生 Race Condition（競態條件）。這種錯誤並非每次都出現，視排程而定，極難偵錯。

核心：讀取→修改→寫入

Race Condition 通常發生在以下三個步驟的循環中：

STEP 1
READ
從記憶體讀取目前的數值
STEP 2
MODIFY
在 CPU 中進行運算（例如 +1）
STEP 3
WRITE
將結果存回記憶體

當兩個執行緒同時進行這三個步驟，順序可能會交錯，導致其中一個執行緒的修改被另一個執行緒「覆蓋」。

生活化的例子：銀行帳戶

假設帳戶餘額為 $1,000，兩筆交易（A 和 B）同時發生，都要存款 $100：

時間執行緒 A（存款 $100）執行緒 B（存款 $100）帳戶餘額
T1READ：$1,000—$1,000
T2—READ：$1,000$1,000
T3MODIFY：1000+100=1100—$1,000
T4WRITE：$1,100—$1,100
T5—MODIFY：1000+100=1100$1,100
T6—WRITE：$1,100 （覆蓋！）$1,100

🔴 最終結果：被領走了 $200，但餘額只增加了 $100

  • 帳戶本應有 $1,200，實際只有 $1,100
  • $100 的存款就這樣消失了，沒有人拿走了它
  • 這就是 Race Condition 造成的資料不一致

如何解決 Race Condition？

核心目標是實作互斥（Mutual Exclusion），確保同一時間只有一個執行緒能進入「臨界區」。

🔐

Lock / Mutex（鎖）

執行緒 A 要操作資料時先上鎖，執行緒 B 必須等 A 解鎖後才能進入。

⚡

Atomic Operations（原子操作）

將「讀取→修改→寫入」封裝成一個不可分割的操作，硬體保證執行期間不會被中斷。

🚦

Semaphore（信號量）

用來控制同時存取特定資源的執行緒數量，可靈活支援多執行緒同時讀取的場景。

為什麼它很危險？

🔍 極難 Debug

  • Race Condition 並非每次執行都會發生，取決於作業系統的排程和處理器速度
  • 可能在開發環境運作正常，但在高負載的正式環境才突然崩潰
  • 錯誤行為可能每 1000 次才出現一次，幾乎無法重現

⚠️ 安全性漏洞

  • 駭客有時會利用極短的時間差（例如檢查權限與執行指令之間）來繞過系統的安全機制
  • 稱為 TOCTOU 攻擊（Time-of-Check to Time-of-Use）
