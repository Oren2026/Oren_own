# Race Condition（競態條件）

## 定義

Race Condition（競態條件）是多工處理或平行運算環境中，一種常見且棘手的軟體錯誤。當多個進程（Process）或執行緒（Thread）**同時存取並試圖修改共享數據**，而最終的結果**取決於這些操作執行的先後順序**時，就會發生 Race Condition。

簡單來說，這就像是多個人同時衝向同一個目標，誰先到達、誰後到達會決定最終的結局，而這種不確定性會導致系統出現無法預期的行為。

---

## 1. 核心觀念：讀取與寫入的時差

Race Condition 通常發生在以下三個步驟的循環中（**讀取-修改-寫入**）：

1. **Read（讀取）**：從記憶體拿到目前的數值。
2. **Modify（修改）**：在 CPU 中進行運算（例如加 1）。
3. **Write（回寫）**：將結果存回記憶體。

當兩個執行緒同時進行這三個步驟，順序可能會交錯，導致其中一個人的修改被另一個人「覆蓋」。

---

## 2. 生活化的例子：銀行帳戶提款

假設你有一個銀行帳戶，餘額為 $1,000。現在有兩筆交易（A 和 B）同時發生，都要提款 $100：

| 時間 | 執行緒 A (提款 $100) | 執行緒 B (提款 $100) | 帳戶餘額 (記憶體) |
|------|---------------------|---------------------|------------------|
| T1 | 讀取餘額：$1,000 | | $1,000 |
| T2 | | 讀取餘額：$1,000 | $1,000 |
| T3 | 計算新餘額：1000 - 100 = 900 | | $1,000 |
| T4 | 回寫餘額：$900 | | $900 |
| T5 | | 計算新餘額：1000 - 100 = 900 | $900 |
| T6 | | 回寫餘額：$900 | $900 |

**最終結果：帳戶被領走了 $200，但餘額卻只扣了 $100。** 這就是典型的 Race Condition 造成的資料不一致。

---

## 3. 如何解決 Race Condition？

要防止這種情況，核心目標是實作**互斥（Mutual Exclusion）**，確保在同一時間只有一個執行緒能進入「臨界區（Critical Section）」。常見的方法包括：

- **Lock / Mutex（鎖）**：當執行緒 A 要操作數據時先「上鎖」，執行緒 B 必須等待 A 解鎖後才能進入。
- **Atomic Operations（原子操作）**：將「讀取-修改-寫入」封裝成一個不可分割的操作，保證在執行期間不會被中斷。
- **Semaphore（信號量）**：用來控制同時存取特定資源的執行緒數量。

---

## 4. 為什麼它很危險？

- **難以調試（Debug）**：Race Condition 並非每次執行都會發生。它取決於作業系統的排程和處理器速度。可能在開發環境運作正常，但在高負載的正式環境才突然崩潰。
- **安全性漏洞**：駭客有時會利用極短的時間差（例如檢查權限與執行指令之間）來繞過系統的安全機制。

## 總結
