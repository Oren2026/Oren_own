Part 1 · Process / 程序

fork() — 影分身術

fork() 是 Unix 的「影分身」。呼叫一次、返回兩次——父程序拿到的返回值是子 PID，子程序拿到的返回值是 0。作業系統把整個父程序的記憶體空間完整複製一份，從此兩者各走各路。
函式 功能 說明
fork() 影分身，複製一份行程 呼叫一次、返回兩次（父拿子PID = 正整數，子拿 0）
execvp() 換魂，变成另一個程式 不開新行程，在現有行程上替換成新程式映像
wait() 父等子，幫子收屍 子行程結束後若父沒呼叫 wait，會變成僵尸（Zombie）
waitpid() 指定等某一個子 比 wait() 更精確，可設定 WNOHANG 不阻塞
system() 包好的 fork+exec+wait 一行執行任意 shell 命令，簡易但有安全風險

⚠ Zombie 產生條件
子行程結束 → 行程表中仍有一筆記錄（PCB）→ 父行程還沒呼叫 wait() → 子行程進入 Zombie 狀態。累積太多 Zombie 會佔滿行程表。
Part 2 · File System / 檔案系統

檔案操作 — 系統層 vs 標準層

Unix 把幾乎所有東西都抽象成「檔案」——硬碟檔案、鍵盤輸入、螢幕輸出、網路連線，全部都是檔案描述符（file descriptor）。理解 FD 0/1/2，等於理解 I/O Redirection 的鑰匙。
函式 層次 功能
open() 系統層 開啟檔案，拿到一個 FD（檔案描述符，整數）
read() 系統層 用 FD 讀取資料，一次讀固定位元組
write() 系統層 用 FD 寫入資料
close() 系統層 關閉 FD，釋放系統資源
gets() / puts() 標準層 C 標準庫，讀寫字串指標，無邊界檢查（不安全）
fgets() / fputs() 標準層 安全版，指定最大讀取長度（buffer size）

I/O Redirection — 插進 0 · 1 · 2

檔案描述符 0、1、2 是標準輸入、輸出、錯誤輸出。I/O Redirection 的原理很簡單：用 open() 開啟檔案，把拿到的 FD 複製到 0、1、2 的位置，從此程式的 stdin/stdout/stderr 就自動送到檔案了。

mmap() — 檔案映射到記憶體

mmap() 把檔案直接映射到行程的虛擬記憶體空間。從此讀記憶體就是在讀檔案，寫記憶體就是在寫檔案——完全繞過 read()/write()。適合大檔案處理與共享記憶體（shared memory）。
💡 mmap vs read/write
read(fd, buf, n) → kernel 拷貝磁碟資料到 page cache，再拷貝到用戶空間（兩次拷貝）。
mmap(fd, ...) → 用戶空間直接映射到 page cache（零拷貝，適合大檔案）。

socket — 網路版的檔案

socket() 是「網路檔案」。Unix 設計哲學：幾乎所有 I/O 都是「開檔 → 讀/寫 → 關檔」。socket() 打開一個網路端點，之後用 read()/write() 收發網路資料，跟操作一般檔案完全相同。
socket() 開網路檔案 建立網路端點，回傳一個 FD
bind() 綁定位址 把 socket FD 綁定到 IP + Port（如 80）
listen() 監聽連線 開始接受客戶端連線（變成 server socket）
accept() 接受連線 取出排隊中的客戶端，建立真正的通訊 socket
connect() 發起連線 client 端連到 server
read/write() 網路傳輸 和操作檔案完全相同，讀寫網路資料

💡 epoll 的價值
傳統 select() 需要輪詢所有 FD，FD 多的時候效率低。epoll() 採用事件驅動——作業系統只通知「哪些 FD 有事件」，不用全量掃描。這就是 Nginx、Redis 高效能的祕密之一。
