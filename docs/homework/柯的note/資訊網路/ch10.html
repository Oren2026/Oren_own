📡 第 10 章｜ARP · RARP · ICMP

網路層的重要協定不僅於 IP 協定。本章探討 ARP、RARP 與 ICMP，三者都是 TCP/IP 協定的一部分，是了解網路層運作不可或缺的知識。

🔗 1｜簡介

ARP 機制

ARP（Address Resolution Protocol，位址解析協定）
將 IP 位址（邏輯位址）解析為 實體位址（MAC 位址）。

RARP 機制

RARP（Reverse Address Resolution Protocol，反向位址解析協定）
將 實體位址（MAC 位址）解析為 IP 位址（邏輯位址）。

ICMP 機制

ICMP（Internet Control Message Protocol，網際網路控制訊息協定）
用於錯誤回報、偵錯與控制，是網路層的輔助協定，需與 IP 標頭搭配使用。

ARP 與 RARP — 密實體位址與邏輯位址

網路設備同時擁有兩種位址：
  • 實體位址（MAC）：燒錄在網卡中，全域唯一，由 IEEE 管理分配。
  • 邏輯位址（IP）：由網路管理員或 DHCP 動態分配，可改變。
ARP 負責 IP → MAC 的對應查詢；RARP 則反向處理。

🔗 2｜ARP 機制

ARP 核心原理：傳送資料前，會先檢查 ARP Cache（ARP 對照表）。若找不到對應的 IP 位址，才廣播 ARP 請求。

運作流程圖

假設主機 A（140.134.36.56）要和主機 B（140.134.36.110）通訊：

  • 主機 A 先查自己的 ARP Cache，找不到 140.134.36.110
  • 主機 A 廣播 ARP 請求（目標 MAC 為 FF-FF-FF-FF-FF-FF）
  • 所有主機收到，但只有 140.134.36.110 回覆 ARP 回應
  • 主機 A 收到回覆後，將 140.134.36.110 與其 MAC 位址寫入 ARP Cache

ARP 請求與回覆訊息傳送

訊息三層封裝：
資料連結層標頭網路層標頭訊息內容
含目標 MAC（廣播為 FF-FF-FF...）含發送端 IPARP 請求/回覆資料

🔄 3｜RARP 機制

RARP 應用場景：來源端知道自己 MAC 位址，但不知道 IP 位址。
必須倚賴 RARP 伺服器執行查詢工作。

常見於：無磁碟機或終端機的場合——電腦沒有硬碟，無法儲存 IP 位址，開機時需向 RARP 伺服器查詢自己的 IP。

運作流程

  • 無 IP 主機廣播 RARP 請求（含自己的 MAC 位址）
  • RARP 伺服器收到，查表後回覆 RARP 回應（含對應 IP 位址）
  • 主機取得 IP 位址，即可正常通訊

RARP 與 ARP 的比較

ARPRARP
查詢方向IP → MACMAC → IP
需求已知 IP，查 MAC已知 MAC，查 IP
伺服器不需要需要 RARP 伺服器
常見用途一般主機通訊前解析無碟主機開機取得 IP

📦 4｜ARP/RARP 標頭

ARP 與 RARP 的標頭定義相同，共分 9 個欄位，每欄位各有其長度與用途。

標頭欄位格式（9 欄位）

欄位長度說明
Hardware Type16 位元硬體類型，如 Ethernet = 1（0001）
Protocol Type16 位元協定類型，如 IP = 0800ₕ
HLEN8 位元硬體位址長度（位元組），如 Ethernet = 6
PLEN8 位元網路協定位址長度（位元組），如 IP = 4
Operation16 位元操作型態（見下表）
Sender HA可變發送端硬體位址（MAC）
Sender IP可變發送端邏輯位址（IP）
Target HA可變目的端硬體位址（MAC）
Target IP可變目的端邏輯位址（IP）

Operation 欄位值

數值操作
1ARP 請求（ARP Request）
2ARP 回覆（ARP Reply）
3RARP 請求（RARP Request）
4RARP 回覆（RARP Reply）

Target HA 特殊規則（ARP 請求時）

  • ARP 請求時，Target HA 欄位值為 00-00-00-00-00-00
  • 資料連結層廣播目的端位址為 FF-FF-FF-FF-FF-FF
  • 回覆時，Target HA 即為請求端真實 MAC 位址

📡 5｜ICMP 協定

ICMP（Internet Control Message Protocol）：網際網路控制訊息協定。

IP 協定主要在「盡力傳送」封包，但傳送過程中遭遇困難與問題（網路擁塞、主機故障、主機未開機等），並非 IP 協定所能解決。

ICMP 便是用來處理此項工作的機制，讓使用者或路由器依需要進行偵錯與控制。

屬於網路層的通訊協定，必需與 IP 標頭搭配，無法單獨運作。

常見應用

  • ping — 發送 ICMP Echo Request，偵測主機是否可達
  • tracert（Windows）/ traceroute（Linux）— 追蹤封包經過的路徑
  • 路由器回報傳輸錯誤（如目的端不可達）

ICMP 訊息類型一覽

Type類型名稱
0回聲回覆（Echo Reply）
3目的端無法抵達（Destination Unreachable）
4來源端抑制（Source Quench）
5路徑重新定向（Redirect）
8回聲請求（Echo Request）
11傳輸逾時（Time Exceeded）
12參數問題（Parameter Problem）
13/14時戳請求/回覆（Timestamp Request/Reply）
15/16網路位址資訊請求/回覆（Information Request/Reply）
17/18網路遮罩請求/回覆（Address Mask Request/Reply）

🔧 6｜ICMP 標頭格式

無論哪種 ICMP 訊息類型，標頭前面 32 位元的格式都相同。

三大基本欄位

欄位長度說明
Type8 位元ICMP 訊息類型（如 0、3、8、11 等）
Code8 位元內容代碼，對類型發生原因做進一步說明
Checksum16 位元檢驗和，驗證訊息完整性

查詢訊息標頭格式

Type (8)Code (8)Checksum (16)
Identifier (16) ｜ Sequence Number (16)
  • Identifier 與 Sequence Number 各 16 位元
  • 請求與回覆時這兩個欄位數值相同，以便相互辨識

錯誤訊息標頭格式

Type (8)Code (8)Checksum (16)
未定義或其他 (32)
IP 標頭 + 64 位元的原始資料
  • 前面三個欄位後的 32 位元：有些類型有定義，有些未定義（以 0 填充）
  • 攜帶原始 IP 標頭與 64 位元資料，以便知道錯誤訊息屬於哪個原始封包

Echo Request / Echo Reply

時戳請求與回覆（Timestamp Request/Reply）

  • ICMP Type：13（請求）/ 14（回覆）
  • Code 欄位皆為 0
  • 以 毫秒（ms）為單位，依格林威治時間為準
  • 主要功能：測量訊號在主機間的傳輸延遲、查詢網路主機時間、系統時間同步調整

時戳訊息格式

Type (8)Code (8)Checksum (16)
Identifier (16) ｜ Sequence Number (16)
原始時戳 Original Timestamp (32)
接收時戳 Receive Timestamp (32)
傳送時戳 Transmit Timestamp (32)

單趟延遲計算範例

單趟延遲時間 = (接收時間 - 原始時間 - 傳送延遲) / 2
                = (122000 - 120300 - 300) / 2
                = 700 ms

⚠️ 8｜Destination Unreachable

  • ICMP Type = 3 訊息，屬於 ICMP 錯誤訊息
  • 無法抵達目的節點的原因有 16 種情況（Code 0~15）

無法抵達目的端原因（Code 欄位）

Code原因
0目的端網路無法連接
1目的端主機無法連接
2指定的通訊協定不存在
3指定的連接埠不存在
4資料需被分割並設定不可分割位元
5來源選擇路徑失敗
6目的端網路無法辨識
7目的端主機無法辨識
8來源端主機被隔離
9禁止與目的端網路通訊
10禁止與目的端主機通訊
11拒絕此網路服務類型

13被過濾禁止通訊
14違反主機優先權
15優先權實行中斷

🛡️ 9｜其他 ICMP 訊息類型

來源端抑制（Source Quench）— Type 4

  • 屬於 ICMP 錯誤訊息
  • 功能：對網路流通容量進行控制，修正網路資料傳送量
  • Code 欄位值：0
類型 (8)代碼 (8)檢驗和 (16)
未定義 (32)：0
IP 標頭 + 64 位元的原始資料

路徑重新定向（Redirect）— Type 5

  • 屬於 ICMP 錯誤訊息
  • Code 欄位說明：
    Code意義
    0因網路變動而更改傳輸路徑
    1因主機變動而更改傳輸路徑
    2因網路和服務類型的變動而更改傳輸路徑
    3因主機和服務類型的變動而更改傳輸路徑
類型 (8)代碼 (8)檢驗和 (16)
Gateway IP 位址 (32)
IP 標頭 + 64 位元的原始資料

傳輸逾時（Time Exceeded）— Type 11

  • 屬於 ICMP 錯誤訊息
  • 可藉 IP 標頭 TTL 欄位值判斷
  • Code 欄位：
    Code意義
    0因迴圈問題造成逾時傳輸（TTL=0）
    1因封包重組超過設定時間而造成逾時傳輸

參數問題（Parameter Problem）— Type 12

  • 屬於 ICMP 錯誤訊息
  • Code = 0：某欄位值發生錯誤，發生位置由指標（Pointer）欄位說明（指向錯誤位元組）
  • Code = 1：標頭缺少某資料選項，但無法說明缺少的項目為何（Pointer 欄位以 0 填充）
類型 (8)代碼 (8)檢驗和 (16)
指標 Pointer (8) ｜ 未定義 (24)：0
IP 標頭 + 64 位元的原始資料

網路位址資訊請求/回覆（Information Request/Reply）— Type 15/16

  • 屬於 ICMP 查詢訊息
  • 功能：向位址伺服器查詢自己的 IP 位址（類似 RARP）
  • 此類訊息幾乎已為 RARP 或 BOOTP 所取代，現在很少使用
類型 (8)代碼 (8)檢驗和 (16)
識別碼 (16) ｜ 序列碼 (16)
位址資訊 (32)

網路遮罩請求/回覆（Address Mask Request/Reply）— Type 17/18

  • 屬於 ICMP 查詢訊息
  • 功能：向閘道查詢網路主機的子網路遮罩位址（Subnet Mask）
類型 (8)代碼 (8)檢驗和 (16)
識別碼 (16) ｜ 序列碼 (16)
子網路遮罩位址 (32)
