OSI 通訊七層與封裝概念

OSI 封裝 TCP UDP Header

OSI 通訊七層模型

OSI（Open Systems Interconnection）模型把網路通訊分成 7 層，由上而下：

應用層 Application (L7)
展示層 Presentation (L6)
會議層 Session (L5)
傳輸層 Transport (L4)
網路層 Network (L3)
資料鏈結層 Data Link (L2)
實體層 Physical (L1)

記憶口诀：All People Seem To Need Data Processing（請、請、...）

傳送端由上往下包裝（封裝），接收端由下往上解開（解封裝）。每一層只跟相鄰的上下層溝通。

封裝（Encapsulation）過程

當資料要傳送時，每經過一層就會加上該層的 Header（標頭），有時還會加上 Trailer（標尾）：

📦 封裝層級（由上往下）
  • Data（資料）：應用層產生的實際訊息
  • Segment（段落）：傳輸層加上 TCP/UDP Header
  • Packet（封包）：網路層加上 IP Header
  • Frame（框架）：資料鏈結層加上 MAC Header + Trailer（FCS）
  • Bit（位元）：實體層轉為 0/1 訊號
Layer 4 — 傳輸層
TCP / UDP Header + Data = Segment
確認來源/目的連接阜、流量控制、錯誤檢查等
Layer 3 — 網路層
IP Header + Segment = Packet
確認來源/目的 IP 位址、路由轉送
Layer 2 — 資料鏈結層
MAC Header + Data + Trailer = Frame
MAC 位址定址、Frame 同步、FCS 錯誤偵測

Frame 與 封包 的差異

Frame Packet（封包） Segment
層級 L2 資料鏈結層 L3 網路層 L4 傳輸層
定址方式 MAC 位址（實體位址） IP 位址（邏輯位址） Port（連接阜）
控制資訊 Header + Trailer（FCS） 主要 Header TCP 或 UDP Header
交換方式 Switch（交換機） Router（路由器） 端點主機

Header 標頭 與 Trailer 標尾

每一層在封裝時，會在資料的前面加上該層的 Header，有些層也會在後面加上 Trailer。

Header
位於封裝資料的最前面
攜帶該層的控制資訊（來源、目的、協定類型、長度等），讓接收端同一層能解讀。
Trailer
位於封裝資料的最後面（L2 為主）
通常用於錯誤偵測（如 Ethernet 的 FCS）。放在最後，因為接收端可以從頭快速檢查到結尾。
🔍 為什麼分層的 Header 很重要？

每一層只解讀自己的 Header，讀完後就把該層的 Header + Trailer 剝除，交給下一層處理。這樣各層可以獨立運作、替換、更新。

TCP 與 UDP 的比較

特性 TCP UDP
連線類型 連接導向（Connection-oriented） 無連線（Connectionless）
可靠性 可靠 — 有確認、重傳機制 不可靠 — 不保證送達
流量控制 ✅ 有（Sliding Window） ❌ 沒有
順序控制 ✅ 有（Sequence Number） ❌ 沒有
速度 較慢（額外開銷大） 較快（輕量）
Header 大小 20~60 bytes（可變） 固定 8 bytes
應用場景 網頁、郵件、檔案傳輸 影片串流、遊戲、DNS
TCP 像掛號信（有回執、可追蹤），UDP 像明信片（寄出後不管）。選擇哪個取決於應用需要可靠性還是速度。
TCP Header 結構（重點欄位）
主要欄位說明：
Source Port 來源連接阜（發送端的應用程式）
Dest Port 目的連接阜（接收端的應用程式）
Seq Number 序列號 — 確保資料順序正確
Ack Number 確認號 — 告訴發送端收到了多少資料
Flags 控制旗標：SYN（連線請求）、ACK（確認）、FIN（結束連線）、RST（重置）
Checksum 錯誤檢查欄位
UDP Header 結構
比 TCP 簡單很多，只有 4 個欄位，總大小固定 8 bytes：
Source Port 來源連接阜（可為 0 表示不需要回覆）
Dest Port 目的連接阜
Length 整個 UDP 封包的長度
Checksum 錯誤檢查（可選，IPv4 可關閉）
